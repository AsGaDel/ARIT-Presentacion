06 · Backend · estructura interna

apps/<nombre>/

Estructura interna de una app Django.

models.py Define las tablas de la base de datos.

Cada clase Python es una tabla. Aquí vive Incident con sus campos título, descripción, latitud y estado.

serializers.py El traductor entre Python y JSON.

Convierte un objeto Incident en JSON para el frontend, y valida el JSON que llega de vuelta.

views.py Decide qué hacer con cada petición.

Aquí está la lógica: “dame los pendientes”, “crea uno nuevo”, “márcalo como resuelto”.

urls.py La centralita de direcciones.

Conecta cada URL con su vista. /api/incidents/3/ entra por aquí antes de llegar al código.

permissions.py Quién puede hacer qué.

Un ciudadano puede comentar; un trabajador cambiar estado; un admin borrar. Se aplica antes de la vista.

filters.py Búsquedas automáticas desde la URL.

?status=pending&priority=Crítico se traduce solo en una consulta a la base de datos.

07 · Backend · models.py

apps/incidents/models.py

El modelo de datos.

class Incident(models.Model):
    class Status(TextChoices):
        PENDING     = 'pending',     'Pendiente'
        IN_PROGRESS = 'in_progress', 'En proceso'
        RESOLVED    = 'resolved',    'Resuelto'

    title       = CharField(max_length=200)
    reporter    = ForeignKey(User, on_delete=CASCADE)
    status      = CharField(choices=Status.choices)
    latitude    = DecimalField(9, 6)
    longitude   = DecimalField(9, 6)

    # Evita un COUNT(*) en cada listado
    vote_count  = PositiveIntegerField(default=0)

    # Soft delete: conserva el historial
    deleted_at  = DateTimeField(null=True, blank=True)

Lo que aporta

  • TextChoicesenum integrado con Django: status pasa a ser un campo con opciones validadas.
  • ForeignKey con CASCADEsi se borra el usuario, sus incidencias se eliminan en cadena.
  • DecimalField(9,6)precisión de unos 10 cm para coordenadas geográficas.
  • vote_count desnormalizadose actualiza al votar; evita contar votos en cada listado.
  • deleted_at en lugar de DELETEborrado lógico, el historial se conserva.

08 · Backend · serializers.py

apps/incidents/serializers.py

El traductor entre Python y JSON.

class IncidentSerializer(ModelSerializer):
    photos        = IncidentPhotoSerializer(many=True)
    is_voted      = SerializerMethodField()
    is_subscribed = SerializerMethodField()
    reporter_email = EmailField(
        source='reporter.email', read_only=True,
    )

    def get_is_voted(self, obj):
        request = self.context.get('request')
        return obj.votes.filter(
            user=request.user,
        ).exists()

    def to_representation(self, instance):
        data = super().to_representation(instance)
        # Oculta campos internos a no-admin
        if request.user.role != 'admin':
            data.pop('admin_notes', None)
        return data

Lo que aporta

  • ModelSerializerderiva el JSON directamente del modelo, sin escribir cada campo.
  • SerializerMethodFieldañade campos calculados: is_voted, is_subscribed se evalúan por usuario.
  • source=expone reporter.email como un campo plano del JSON.
  • to_representationhook para esconder admin_notes y otros campos a no-admins. Una sola clase, salida distinta por rol.

09 · Backend · views.py

apps/incidents/views.py

La lógica de negocio.

class IncidentViewSet(viewsets.ModelViewSet):
    serializer_class = IncidentSerializer
    filterset_class  = IncidentFilter

    def get_queryset(self):
        qs = Incident.objects.select_related('reporter') \
            .prefetch_related('photos', 'votes')
        # Los admins ven incluso los borrados
        if self.request.user.role != 'admin':
            qs = qs.filter(deleted_at__isnull=True)
        return qs

    @action(detail=True, methods=['post'])
    def vote(self, request, pk=None):
        incident = self.get_object()
        # Toggle: si existe, borra; si no, crea
        vote, created = Vote.objects.get_or_create(
            incident=incident, user=request.user,
        )
        if not created: vote.delete()
        return Response({'voted': created})

Lo que aporta

  • ModelViewSetlos 5 endpoints CRUD (list, retrieve, create, update, destroy) salen automáticos.
  • get_queryset filtrado por rollos admins ven todo, los demás no ven incidentes borrados.
  • select_related + prefetch_relatedevita el problema N+1: un solo SQL en vez de uno por incidente.
  • @action añade endpoints custom/incidents/{id}/vote/, /subscribe/, /photos/… al mismo router.

11 · Notificaciones en tiempo real

Django Channels + Redis

Notificaciones en tiempo real con WebSockets.

El problema

En una web normal el navegador pregunta y el servidor responde, pero el servidor no puede avisar al navegador por su cuenta cuando algo cambia.

La solución

WebSockets: una línea telefónica permanente entre navegador y servidor. La línea queda abierta y, cuando hay novedad, el servidor llama directamente al cliente.

  • Django Channels es la extensión que permite a Django mantener esas conexiones abiertas.
  • Redis hace de intermediario: reparte los mensajes al canal correcto.

React + Vite + Tailwind

Tecnologías del frontend.

Tres herramientas, un objetivo: que cargue rápido y se vea consistente.

  • React   construye la web como piezas de Lego reutilizables (botones, tarjetas, formularios).
  • Vite   es el motor de desarrollo: cada vez que guardamos un archivo, el navegador se actualiza al instante.

13 · Frontend · cliente HTTP

src/api/axiosConfig.js

Axios: una sola puerta para hablar con la API.

const api = axios.create({
  baseURL: '/api',
  headers: { 'Content-Type': 'application/json' },
})

// 1· Añade el JWT a cada petición saliente
api.interceptors.request.use((config) => {
  const token = localStorage.getItem('access_token')
  if (token)
    config.headers.Authorization = `Bearer ${token}`
  return config
})

// 2· Si una respuesta vuelve 401, refresca y reintenta
api.interceptors.response.use(
  (r) => r,
  async (error) => {
    if (error.response?.status === 401 && !original._retry) {
      original._retry = true
      const { data } = await axios.post(
        '/api/auth/token/refresh/', { refresh },
      )
      localStorage.setItem('access_token', data.access)
      return api(original)  // reintenta
    }
    return Promise.reject(error)
  },
)

Lo que aporta

  • Una instancia compartidatodas las llamadas pasan por aquí con baseURL /api.
  • Interceptor de peticiónañade el JWT automáticamente. Ningún componente tiene que acordarse del token.
  • Interceptor de respuestasi el backend devuelve 401, pide un access nuevo con el refresh y reintenta la petición original.
  • El usuario no ve “sesión expirada”mientras el refresh siga vivo, la app vuelve a la actividad sin volver a pedir login.
